# 团队组织与分工

> 本文定义「摇人吧」（OpenRobotService）项目的团队构成、协作模式与分工。
> 产品形态见 [PRODUCT.md](./PRODUCT.md)，技术架构见 [ARCHITECTURE.md](./ARCHITECTURE.md)。

## 一、团队构成（5 人）

| 成员 | 岗位 |
|------|------|
| 总负责人 | 总负责 + 商业化推广运营 |
| 产品经理 | 产品规格 + 后台管理模块 + 运营 |
| 后端工程师 | 核心层 + 系统任务模块 |

## 二、核心协作模式

### 1. 100% AI 生成代码

代码由 AI 生成，因此每个成员的真实角色是所负责模块的 **产品 + 架构 + 质检 owner**——核心职责是**设计模块、指挥 AI 生成、验证正确性**，而非亲手敲代码。岗位名称只代表"擅长加权"，不是能力边界：例如产品经理也能凭 AI 拥有"后台管理"整个模块（他定义看什么指标，AI 把仪表盘生成出来）。

### 2. 功能垂直切分

不按技能横切（前端做所有页面、后端做所有接口），而是**按功能垂直切**：每人独立负责一个功能模块的全栈（H5 + 后端 API + 对应 AI Agent）的设计、开发与验证。

### 3. 滚动递进，不硬分阶段

工作分两类——**基础框架（共享底座）**与**功能模块（垂直切片）**。两者**滚动式推进、并行交织**：地基先行，但不必等底座全部完工才动功能模块；随着接口契约逐步就绪，垂直模块陆续铺开。

> 底座是所有功能模块的"地基 + 宪法"。底座阶段让岗位回归传统本行共建，既造出地基，又让团队在共建中对地基达成共识、磨合协作——避免日后所有理解都依赖总负责人单点传递。

## 三、分工总表（三维度）

每位成员的工作落在三个维度：**① 基础框架**（按传统岗位共建底座）、**② 功能模块**（垂直 owner 一个切片）、**③ 商业运营**（总负责人主导，各成员从自身专业贡献支点）。

| 成员 | ① 基础框架（共建底座） | ② 功能模块（垂直 owner） | ③ 商业运营（共同参与） |
|------|------|------|------|
| **总负责人** | 架构契约拍板、边界裁决（轻度技术参与） | 关键节点集成把关 | **主轴**：服务号运营/认证拉新、开源社区运营、商业模式、行业资源对接、知识库行业燃料 |
| **产品经理** | 产品规格：数据字典 / 接口契约 / 页面规范（文档） | **后台管理**（看板/项目/风险/日报） | 运营内容策划、用户增长方案、数据看板支撑决策、案例与文案 |
| **后端工程师** | 核心层（配置/DB/**数据模型**/认证 RBAC）+ 微信外壳 | **系统任务**（统一任务收件箱·业务最重） | 服务号技术对接（模板消息/菜单/支付预留）、数据埋点与统计后端 |
| **前端工程师** | 前端基建（Vue+Vant 脚手架/设计系统/加载页/路由） | **我要摇人**（报障/AI 咨询/我的工单） | 品牌视觉落地、H5 分享传播优化、落地页 |
| **AI 算法工程师** | AI 基础层（LLM 统一接口/RAG/上下文）+ 知识库骨架 | **三个 Agent** + 知识库内容 | AI 能力卖点包装、知识库护城河展示、智能化 demo |

## 四、总负责人 = 双主轴

总负责人不陷入具体编码，而是把控两条主轴：

1. **对内**：把控架构契约与方向，做边界裁决与关键节点集成把关（轻度技术参与，拍板为主）。
2. **对外（主战场）**：全力做**商业化推广运营**——这是"官方服务「摇人吧」是门面、开源版是主线"定位下的生死线，且只有总负责人有视野与资源去做：
   - 服务号注册认证、内容运营、拉新
   - 开源社区运营（GitHub 曝光、文档、贡献者、星标——开源主线的获客渠道）
   - 商业模式设计（开源免费 + 增值变现）
   - 行业资源对接（机器人公司、USP 调度平台合作）
   - 知识库的行业知识来源（护城河的"燃料"）

## 五、商业化的两大增长引擎

商业运营由总负责人主导，但**最有效的增长来自两个工程引擎**，由对应成员负责：

- **前端「分享裂变」**：H5 分享卡片、落地页、传播优化——决定产品能否在微信生态里裂变扩散。
- **AI「卖点包装」**：智能诊断、专业知识库——决定用户是否一眼觉得"这个有 AI、很专业"。

这两者比单纯的运营推文更能带来真实增长。

## 六、仓库与代码组织

- **单仓（monorepo）**：全部代码放一个 GitHub 仓库 `OpenRobotService`。因为产品是单体后端 + 单 H5 前端、5 人小团队、模块共享数据模型/认证/组件库、前后端契约需原子化同步、AI 生成需全局上下文、开源用户一次 clone 即得。**不拆多仓**（内部 ProjectPlatform 拆 5 仓导致共用库、硬编码互调的痛点正是要收敛掉的）。
- **目录即模块，CODEOWNERS 分权**：用目录划分模块所有权，`.github/CODEOWNERS` 界定"改某目录须对应 owner 评审"，既分权又不割裂。将来某部分真需独立演进/部署时再拆出去（先单仓，拆出去易、合回来难）。
- **微信代码无特殊处理**：`backend/app/wechat/` 与其他后端代码一样正常 git 管理。**不存在"提交代码到微信平台"**——微信公众平台后台只填配置（服务器 URL / Token / AESKey / 域名 / 菜单），不接收代码；服务号 + H5 是部署在自己服务器的普通网页（小程序才需上传代码）。
- **私有敏感信息靠 `.env` 分离（方案 A）**：代码 100% 开源，真实微信密钥 / 知识库数据 / 运营配置只存在于生产服务器的 `.env`（已 gitignore）。开源仓只含 `.env.example` 占位。将来私有配置/数据复杂到需版本化时，再加一个私有"部署配置仓"（代码单仓与私有数据另放是两个维度，不冲突）。

### 目录结构

```
backend/app/
├── core/ models/ schemas/   # 后端工程师（底座：配置/数据模型/认证）
├── wechat/                  # 后端工程师（微信外壳）
├── kb/  ai/agents/          # AI 工程师（知识库 + AI 基础层 + 三 Agent）
├── modules/{call,tasks,admin}/  # 三功能模块的后端（call=我要摇人 tasks=系统任务 admin=后台管理）
└── services/               # 跨模块共享业务
frontend/src/
├── shared/ router/ api/     # 前端工程师（设计系统/基建）
└── views/{call,tasks,admin}/    # 三菜单页面
```

## 七、本协作模式的四大风险

100% AI 生成代码 + 垂直并行模式下，以下风险比"谁写代码"更需关注：

1. **接口契约必须先冻结**：垂直并行的前提是底座的数据模型 / API 规范 / 前端组件契约先定。契约是底座共建的核心交付物。
2. **前端一致性**：三个菜单 H5 由不同人分做，必须共用一套设计系统 / 组件库（前端工程师 owner），否则出现三种画风。
3. **「验证」是核心动作**：AI 写代码很快，质量门（测试、评审、联调）才是瓶颈。每个 owner 的真功夫在验收自己模块的正确性，需有统一验证标准。
4. **集成需专人兜底**：5 个模块端到端跑通需有人盯（建议后端工程师或团队共同职责），以把总负责人解放去做商业化。
