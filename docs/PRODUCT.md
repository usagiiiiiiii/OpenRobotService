## 一、产品定位

**一句话**：一个面向工业移动机器人（AGV / AMR）行业的、开箱即用、可自托管的微信服务号平台，覆盖从客户报障、工单流转到项目交付管理的全流程，并由 AI 沿"需求 / 供给 / 管理"三种视角全流程深度参与。

四项根本决策：

| 维度 | 决策 |
|------|------|
| **产品 or 框架** | **产品路线**。OpenRobotService 即「摇人吧」本身，一个开箱即用的成品平台，而非给别人搭台子的抽象框架。 |
| **领域** | **专门面向工业移动机器人**（AGV/AMR），领域绑定，不做通用服务台。 |
| **目标用户** | 三类：①现场工程师 ②客户 ③机器人公司管理人员。全部移动端办公。 |
| **开源关系** | **开源版是主线**。官方服务「摇人吧」是推广重点与门面；内部既有系统未来向开源版收敛。 |

**功能范围 = B + 轻量 C**

- **B（项目交付平台）**：工单（流转/讨论/派单/上报）+ 在线咨询 + 项目管理 + 风险跟踪 + 项目日报 + 授权(license)。
- **轻量 C**：本平台**不**拉取机器人实时遥测；而是由 **USP 调度平台主动上传/导出**两类数据进来——①机器人故障/异常数据 ②每日运行任务统计。平台负责展示、关联工单、供管理人员查看。这是 OpenRobotService 与 USP 调度平台的明确边界。

## 二、核心理念：每个用户都同时具备三种视角

现实中同一个人往往身兼数职：一名工程师既会**提出**需求（报障、提需求），又要**承接**任务（接单处理），还可能**管理**自己负责的项目。按"角色分人"会割裂这种真实工作流，按"行为视角分场景"才贴合本质。

因此，平台的顶层结构不是按角色切分，而是按三种**视角/行为**切分，且对所有用户统一开放：

| 视角 | 行为 | 对应菜单 |
|------|------|---------|
| **需求** | 主动求助 | 我要摇人 |
| **供给** | 被动承接 | 系统任务 |
| **管理** | 监控全局 | 后台管理 |

页面内容则按用户的角色与其在各项目中的身份**动态渲染**——同一个菜单，不同人看到的内容不同。

## 三、微信服务号界面结构

### 底部一级菜单（三个，对所有人统一显示，无二级菜单）

每个菜单直接深链进入一个 H5 页面，内容按角色动态调整。**不使用微信个性化菜单做角色隐藏**——因为人人都有三种视角的需求。

```
微信服务号底部菜单
├─ 🆘 我要摇人   需求视角 · 主动求助
├─ 📥 系统任务   供给视角 · 被动承接
└─ 📊 后台管理   管理视角 · 监控全局
```

### 加载过场页（江湖武侠风）

点击菜单进入 H5、内容加载阶段，各自先显示一个「动画 + 标语」过场页，这是平台的标志性体验：

| 菜单 | 标语 |
|------|------|
| 🆘 我要摇人 | **一支穿云箭，千军万马来相见！** |
| 📥 系统任务 | **系统在手，天下我有！** |
| 📊 后台管理 | **东厂管不了的事，我西厂管！** |

风格要求：
- 三页共用同一套视觉语言（建议毛笔字 / 墨迹 / 卷轴感），成体系。
- **快而不挡路**：动画在数据请求开始时即播，数据返回后立即可淡出/跳过，最长兜底 ~1.5s 后无论如何进内容，网络快时不被动画拖慢。
- 动画淡出后接 Vant 骨架屏（Skeleton），再填真实数据，过渡不突兀。

## 四、三个菜单的功能详解

### 🆘 我要摇人（需求视角）

主动求助入口，面向"我有问题需要解决"的场景。

- **一键报障 / 提交工单**：最高频动作，现场一秒触达。
- **AI 在线咨询**（见第五节）：AI 先引导初步诊断，解决不了再打包转工单。
- **我提交的工单**：列表 → 详情 → 讨论 / 催办 / 升级上报。

### 📥 系统任务（供给视角）

**统一任务收件箱**——凡是"系统要你做的事"都进这里，不限于工单。

- 任务类型：**工单、bug、产品需求**等多种。
- 内容：派给我的任务、待我处理的上报、系统派发的待办（如填日报提醒、巡检/回访）。
- 操作：接单、处理、转派、上报、填写处理结果（AI 辅助生成草稿，见第五节）。

### 📊 后台管理（管理视角）

面向监控与决策，移动端友好的看板，而非 PC 管理台。

- **跨项目看板**：工单总览、风险红黄灯、待审批上报。
- **项目管理**：项目列表 → 详情（含该项目的工单 / 风险 / 日报 / 授权）。
- **机器人数据（轻 C）**：故障/异常列表、每日运行任务统计（数据由 USP 上传）。
- **(admin)** 用户与权限管理。

## 五、AI 在线咨询流程（提单 Agent）

"我要摇人 → 在线咨询"由 AI 全程引导，目标是**减少人工处理量、提升响应及时度**：

```
用户描述问题
     │
     ▼
AI 初步诊断（基于知识库：行业知识 + FAQ + 相似历史案例）
     │
  ┌──┴──────────────┐
  ▼                 ▼
能自助解决        不能解决
  │                 │
引导用户          AI 收集必要信息
自行处理          （补全问题描述、定位、环境等）
  │                 │
  ▼                 ▼
结束             打包生成工单 → 转人工处理
```

AI 不是简单问答，而是**带诊断目标的引导**：能解决就地解决，解决不了则把问题"整理干净"再转人工，让工程师接手时信息已完备。

## 六、角色 → 视图差异速查

| 角色 | 我要摇人 | 系统任务 | 后台管理 |
|------|---------|---------|---------|
| **客户** | 报障、咨询、看自己的工单 | 看与自己相关的回访/确认任务 | 看自己项目的概况 |
| **工程师** | 同上 + 提需求/bug | 派给我的工单/bug、上报、填日报 | 看负责项目的工单/风险 |
| **管理人员** | 同上 | 待我审批的上报、待我派的单 | 跨项目看板、风险概览、机器人统计、决策 |
| **admin** | 全部 | 全部 | 全部 + 用户权限管理 |

> 提交任务、参与讨论是所有角色的通用能力；差异体现在派单、上报接收范围、可见范围与管理权限。

## 七、设计原则小结

1. **三菜单 = 三视角 = 三 AI Agent**，一一对应，AI 沿全流程渗透，非附加功能。
2. **统一显示、内容动态**，贴合"一人多角色"的真实工作流。
3. **纯移动端**，无 PC 后台，管理人员的看板也做成移动 H5。
4. **知识库是护城河**，AI 咨询/辅助的质量取决于知识库的专业度（详见架构文档）。
5. **效果优先于部署轻量**：官方实例是门面，服务效果是生命线。
